# ИБ и ограничения

## Основные вопросы

- Можно ли использовать внешние LLM/API?
- Какие данные запрещено передавать во внешние сервисы?
- Нужна ли on-prem / private cloud модель?
- Какие требования к логированию?
- Какие требования к контролю доступа?
- Нужен ли human-in-the-loop?

## Принцип

Базовый архитектурный принцип проекта:

> **AI inside controlled perimeter** — чувствительные данные не покидают разрешённую среду.

Всё остальное в этом документе — операционализация этого принципа: как классифицировать данные, куда их можно отправлять и по каким правилам сценарий получает допуск.

## Классификация данных (привязка к регуляторике)

Без утверждённой классификации проект нельзя запускать — это входное условие. Ориентир:

|Класс|Что входит|Регулятор / основание|Базовое ограничение для ИИ|
|---|---|---|---|
|**ПДн**|ФИО, контакты, кадровые, клиентские данные|152-ФЗ; ПП РФ №1119 (УЗ); Приказы ФСТЭК №21 и ФСБ №378; 242-ФЗ (локализация)|Только контролируемый контур в РФ; трансграничная передача — по ст. 12 152-ФЗ|
|**Специальные / биометрические ПДн**|здоровье, биометрия|152-ФЗ (ст. 10–11)|Максимальные ограничения; в ИИ — как правило запрет|
|**Коммерческая тайна (КТ)**|ноу-хау, себестоимость, условия сделок|98-ФЗ; внутренний режим КТ|Запрет вне режима КТ; внешние сервисы — нарушение режима|
|**КИИ / значимые объекты**|данные АСУ ТП, РЗА, АСКУЭ|187-ФЗ; Приказы ФСТЭК №235/239|Строгая сегментация; недопущение потоков в универсальную AI-инфраструктуру|
|**Конфиденциальные / ДСП**|внутренние документы, финансовая аналитика|внутренние ЛНА|Контролируемый контур; доступ по ролям|
|**Публичные / согласованные**|пресс-релизы, НПА, открытые источники|—|Допустимы для внешних инструментов|

## Матрица допуска «класс данных × место обработки»

Быстрое правило принятия решения для любого сценария:

|Данные ↓ / Контур →|On-prem / локальная LLM|Российское облако (ДЦ РФ, договор)|Публичный зарубежный LLM|
|---|---|---|---|
|Публичные|🟢|🟢|🟢|
|Конфиденциальные / ДСП|🟢|🟡 (по договору + оценка)|🔴|
|ПДн (обычные)|🟢|🟡 (УЗ, локализация, аттестация)|🔴|
|Спец./биометрия|🟡 (усиленно)|🔴|🔴|
|КТ|🟢 (в режиме КТ)|🟡 (договор + режим КТ)|🔴|
|КИИ (значимые объекты)|🟡 (изолированно)|🔴|🔴|

🟢 допустимо · 🟡 с ограничениями и согласованием ИБ · 🔴 запрещено

## Три нетривиальных требования (легко упустить)

1. **RAG с ПДн = ИСПДн.** Если модель отвечает по внутренней базе, содержащей ПДн, то векторный индекс и хранилище фрагментов — часть ИСПДн. Значит: локализация в РФ (242-ФЗ), соответствующий УЗ (ПП 1119), меры ФСТЭК №21. Вывод для архитектуры: либо держать индекс в контуре, либо исключать ПДн из индексируемого корпуса.
2. **Журналы запросов/ответов — защищаемый актив.** Промпты и ответы часто содержат ПДн/КТ. Сам лог становится объектом защиты: хранение в контуре, доступ по ролям, срок хранения, недопущение выгрузки в аналитические внешние сервисы.
3. **Обезличивание ≠ псевдонимизация.** По 152-ФЗ данные перестают быть ПДн только при **необратимом** обезличивании. Маскирование с возможностью восстановления (в т.ч. по совокупности признаков) не снимает статус ПДн. Оценивать риск реидентификации отдельно.

## КИИ-контур (187-ФЗ) — отдельно

- AI-инструменты **не должны** создавать потоки данных из значимых объектов (АСУ ТП, РЗА, АСКУЭ) в универсальную AI-инфраструктуру.
- Любая интеграция ИИ со значимым объектом — это **изменение объекта** → пересмотр категорирования и мер защиты.
- Сценарии, затрагивающие КИИ, проходят через ответственное лицо по КИИ до пилота.

## Трансграничная передача

Использование публичного **зарубежного** LLM для рабочих данных = трансграничная передача (ст. 12 152-ФЗ) и, для КТ, нарушение режима (98-ФЗ). Для ПДн потребовались бы уведомление РКН и адекватная защита/согласие; на практике для чувствительных данных это **запрет**, а не «согласование».

### CPU-контур Фазы 0 (важное ограничение)

На старте проекта используется временный CPU-контур для проверки UX/спроса (OpenWebUI + Qwen3.5, без GPU — см. [16 - Фаза 0](./16%20-%20%D0%A4%D0%B0%D0%B7%D0%B0%200%20%E2%80%94%20CPU-%D0%BF%D0%B8%D0%BB%D0%BE%D1%82%20%D0%B8%20%D1%84%D0%BE%D0%BA%D1%83%D1%81-%D0%B3%D1%80%D1%83%D0%BF%D0%BF%D0%B0.md)). Этот контур **не аттестован под ПДн/КТ**, поэтому на нём допускаются только публичные и обезличенные данные. Правило доводится до участников фокус-группы под подпись и контролируется ИБ. Продуктивный контур для чувствительных данных строится отдельно, после Gate Ф0, по требованиям матрицы допуска выше.

> Примечание по выбору: «одной лучшей модели» нет — модель подбирается под процесс (тип задачи, объём, чувствительность, латентность). Для жёсткого on-prem практичны GigaChat 3.1 и сильные open-weight (Qwen, DeepSeek). Self-host окупается при стабильно высокой утилизации GPU; при рваной нагрузке облачный API дешевле по TCO.

## Основные риски

|Риск|Влияние|Митигирующие меры|
|---|---|---|
|Утечка ПДн/КТ во внешний сервис|Высокое|Запрет внешних сервисов для чувствительных данных, DLP, локальный контур|
|Галлюцинации модели|Ср./выс.|Human-in-the-loop, RAG со ссылками на источники, проверка в критичных процессах|
|Shadow AI (неконтролируемое использование)|Высокое|Политика использования ИИ, реестр разрешённых инструментов, обучение, техконтроль|
|Ошибки в критичных процессах (финансы, ИБ, GR, юр.)|Высокое|ИИ как ассистент, не автономный исполнитель; обязательная проверка человеком|
|Нарушение регуляторики (ПДн/КТ/КИИ)|Высокое|ИБ- и юр-экспертиза каждого сценария; матрица допуска|
|ПДн в индексе/логах RAG|Высокое|Исключение ПДн из корпуса или контур + УЗ; защита журналов|
|Vendor lock-in|Среднее|Пилот нескольких вариантов, приоритет open-weight и vendor-agnostic оркестрации|

## Паспорт AI-сценария (обязательные поля)

Каждый сценарий описывается по единому шаблону — это то, что ИБ визирует:

- назначение и владелец процесса / владелец данных;
- перечень используемых данных и **класс чувствительности** (по классификации выше);
- место обработки (контур) и обоснование по матрице допуска;
- требования к доступу (RBAC) и логированию (что, где, срок хранения);
- требование к проверке результата человеком (да/нет, где именно);
- оценка риска ИБ и, при необходимости, КИИ/ПДн-экспертиза;
- **решение: можно / нельзя / можно с ограничениями** + условия.
