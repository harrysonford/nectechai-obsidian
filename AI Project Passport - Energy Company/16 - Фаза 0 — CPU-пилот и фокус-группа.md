# Фаза 0 — CPU-пилот и фокус-группа

> **Назначение:** проверить пользовательский интерфейс, сценарии и реальный спрос на ИИ **до** закупки GPU-оборудования.
> **Принцип:** сначала подтверждаем спрос на дешёвом открытом стеке, потом инвестируем в инфраструктуру (Gate Ф0).

## Зачем нужна Фаза 0

Закупка GPU-контура — самая крупная и наименее возвратная статья бюджета (≈22 млн руб., 27% Этапа 1). Разворачивать её «вслепую», не проверив, будут ли сотрудники пользоваться сервисом и удобен ли он, — это инфраструктурный риск. Фаза 0 снимает его: за 6–10 недель на бесплатном открытом стеке и существующих CPU-мощностях мы проверяем UX/UI и спрос, собираем банк рабочих сценариев и только потом принимаем решение о закупке.

## Стек Фазы 0

| Компонент | Что это | Лицензия / стоимость |
|---|---|---|
| **OpenWebUI** | Веб-интерфейс чата с LLM: аутентификация, ролевой доступ, история диалогов, русскоязычный UI, загрузка документов | Открытый код, 0 руб. |
| **Ollama / llama.cpp** | Движок CPU-инференса (без GPU) | Открытый код, 0 руб. |
| **Qwen 3.5** (открытые веса) | Языковая модель, выпуск февраля 2026, лицензия Apache 2.0, поддержка русского языка | Открытые веса, 0 руб. |
| **CPU-сервер** | 2×16 ядер, 128 ГБ ОЗУ, NVMe 2 ТБ | ≈800 тыс. руб. (или ВМ на существующих мощностях — тогда 0) |

## Выбор модели под CPU

Инференс на процессоре медленнее GPU, поэтому модель подбирается под доступную ОЗУ и приемлемую скорость. Лестница вариантов (квантование Q4 ≈ 4 бита на вес):

| Модель | Память (Q4) | Ориентир скорости на серверном CPU | Роль |
|---|---|---|---|
| **Qwen3.5-4B** | ~2,7 ГБ | 15–25 токенов/с | Базовая — гарантированно «летает», отзывчивый UX |
| **Qwen3.5-9B** | ~6 ГБ | 6–12 токенов/с | Целевая — заметно выше качество при приемлемой скорости |
| **Qwen3.5-35B-A3B** | ~20 ГБ ОЗУ | 8–15 токенов/с | **Оптимальная при 128 ГБ ОЗУ.** MoE-архитектура активирует лишь 3 млрд параметров на токен → скорость малой модели при качестве крупной |

> Ключевой момент: **35B-A3B** — это Mixture-of-Experts. Из 35 млрд параметров на каждый токен работают только 3 млрд активных, поэтому модель на CPU идёт сопоставимо с 4B-моделью, но отвечает существенно качественнее. Это лучший вариант для CPU-контура, если сервер располагает ≥128 ГБ ОЗУ. Итоговый выбор — по замерам скорости на целевом сервере.

## Что проверяет Фаза 0 (и что — нет)

**Проверяет:**
- удобство интерфейса и сценариев (UX/UI);
- реальный спрос по подразделениям — кто и как часто пользуется;
- паттерны использования, типовые запросы;
- банк рабочих промпт-сценариев для последующего переноса на продуктив.

**Не проверяет:**
- качество продуктивной модели (оно определяется уже на GPU-контуре — Qwen3.5 на CPU это демонстратор UX, а не эталон качества ответов);
- работу с чувствительными данными (см. ограничение ИБ ниже);
- нагрузочную устойчивость при полном охвате.

## Ограничение ИБ (критично)

CPU-контур Фазы 0 **не проходит аттестацию под ПДн/КТ**. Поэтому:

- на Фазе 0 допускаются **только публичные и обезличенные данные**;
- ограничение фиксируется в правилах фокус-группы и доводится до каждого участника под подпись;
- контроль — со стороны ИБ; при работе с чувствительными данными сессия останавливается.

Это ограничение — не формальность: сотрудник, привыкший к сервису, по инерции может вставить внутренний документ. Правила и контроль обязательны.

## Фокус-группа

- **Состав:** 15–20 сотрудников из подразделений фокуса (ИТ, ИБ, PR, GR, Финансы, Бухгалтерия, Разработка).
- **Нагрузка:** 3–5 одновременных активных сессий; параллелизм ограничивается очередью, контекст 8–16 тыс. токенов.
- **Длительность:** 4–6 недель активного использования.
- **Сбор данных:** анкеты CSAT (удовлетворённость) и SUS (юзабилити), лог сценариев, короткие интервью.

## Критерии Gate Ф0 (решение о закупке GPU)

| Критерий | Порог |
|---|---|
| Удовлетворённость (CSAT) | ≥70% |
| Регулярность использования | ≥60% участников используют сервис ≥3 раз в неделю |
| Банк сценариев | собрано ≥20 рабочих промпт-сценариев |
| Подтверждённый спрос | потребность подтверждена ≥3 подразделениями |

**Исходы Gate Ф0:**

| Решение | Условие | Действие |
|---|---|---|
| Закупать GPU | Критерии достигнуты | Запуск закупки продуктивного контура; перенос сценариев на GPU |
| Аренда вместо закупки | Спрос есть, но CAPEX нежелателен | Аренда GPU в российском облаке (ДЦ РФ, договор) на период MVP |
| Пересмотр | Критерии не достигнуты | GPU **не закупается** (экономия ≈22 млн); подход и сценарии пересматриваются |

При отрицательном исходе максимальные потери ограничены затратами Фазы 0 (≈1,4 млн руб.) и трудозатратами проектной группы до точки решения — это и есть смысл двухступенчатой модели.

## Что дальше (после положительного Gate Ф0)

- CPU-контур сохраняется как тестовая/демонстрационная среда;
- собранные сценарии переносятся на продуктивную GPU-модель;
- **Пилот 3 (PR/GR-мониторинг, публичные данные) может быть запущен уже на CPU-контуре**, не дожидаясь GPU;
- разворачивается RAG-платформа под флагманский Пилот 1.
